# Chronicle — Issue Execution Prompt
## Reusable pattern for completing GitHub issues in VS Code

---

## How this prompt works

This prompt follows a fixed five-phase pattern for every Chronicle issue:
READ → PLAN → BUILD → TEST → CLOSE

Paste this prompt into Claude Code. Fill in the declaration block at the
top. Everything else uses those values automatically — no other substitutions
are needed. Do not skip phases or reorder them.

---

## The Prompt

```
====================================================================
DECLARATION — fill in before running, then do not change
====================================================================

ISSUE_NUMBER:  7
REPO_OWNER:    dgrimesii
REPO_NAME:     Chronicle-MagersCampaign
ISSUE_URL:     https://github.com/dgrimesii/Chronicle-MagersCampaign/issues/7

====================================================================
Everything below this line is generic. Do not edit it.
Wherever you see ISSUE_NUMBER, REPO_OWNER, REPO_NAME, or ISSUE_URL,
substitute the values declared above.
====================================================================

You are working on the Chronicle codebase. Your job is to complete
GitHub issue ISSUE_NUMBER in REPO_OWNER/REPO_NAME by reading it,
planning the work, executing the changes, testing them, and closing
the issue with high-quality notes.

Follow the five phases below in order. Do not begin a phase until
the previous one is complete. Post a GitHub comment at the end of
each phase before moving to the next.

---

PHASE 1 — READ

Step 1a: Read these files before doing anything else:
  - README.md
  - CLAUDE.md
  - Every file mentioned in the issue body (you will discover these
    in Step 1b — read the issue first, then read the files)

Step 1b: Fetch issue ISSUE_NUMBER from REPO_OWNER/REPO_NAME.
Read the full issue body including title, description, acceptance
criteria, labels, milestone, and any existing comments.

Step 1c: Set the issue status to "In Progress" in GitHub.
Do this immediately after reading the issue — before planning begins.
This signals to any collaborators that work is underway.

Step 1d: Post a comment on issue ISSUE_NUMBER:

  ---
  *🤖 This comment was generated by Claude Code*
  ---

  ## Starting work — Phase 1 complete: Read

  **Files read:**
  [list every file you read]

  **Issue understood:**
  [2-3 sentences in plain language: what the issue is asking for,
  why it exists, and what "done" looks like]

  **Blockers or ambiguities:**
  [Any questions or unclear requirements — or "None"]

Do not proceed to Phase 2 until the status is set and the comment
is posted.

---

PHASE 2 — PLAN

Based on what you read in Phase 1, produce a detailed execution plan
derived entirely from the issue body and the files you read.
Do not invent requirements that are not in the issue.

The plan must cover:
  1. Every file that will be created, modified, or deleted — and
     what specifically will change in each
  2. The exact order of changes (some changes depend on others)
  3. The test strategy: what automated tests can be written, what
     must be tested manually, and the pass criteria for each
  4. Any risks — things that could break as a result of this change

Post a comment on issue ISSUE_NUMBER:

  ---
  *🤖 This comment was generated by Claude Code*
  ---

  ## Phase 2 complete: Plan

  **Files to change:**
  | File | What changes | Why |
  |---|---|---|
  [one row per file]

  **Execution order:**
  [numbered list of changes in the order they must happen]

  **Test strategy:**
  | Test | Type (automated/manual) | Pass criteria |
  |---|---|---|
  [one row per test]

  **Risks:**
  [list any breaking risks and how they will be mitigated]

Do not proceed to the gate until the comment is posted.

---

GATE — HUMAN APPROVAL REQUIRED BEFORE PHASE 3

After posting the Phase 2 comment, stop completely.

Print this message to the terminal and wait:

  ✋ PLAN COMPLETE — REVIEW REQUIRED BEFORE BUILD
  ─────────────────────────────────────────────
  The Phase 2 plan has been posted as a comment on:
  ISSUE_URL

  When you have reviewed the plan, reply here with one of:
    "approved"             — proceed with Phase 3 Build
    "revise: [your notes]" — revise the plan before building

  Do not write any code until you receive explicit approval.

Do not begin Phase 3 under any circumstances until the DM types
"approved" in this terminal session. If the DM provides revision
notes, update the plan, post a revised Phase 2 comment on GitHub,
and wait for approval again before proceeding.

---

PHASE 3 — BUILD

Execute the changes exactly as planned in the approved Phase 2 plan.
Derive every change from the issue body and the plan — do not add
scope that was not approved.

Apply CLAUDE.md rules throughout:
  - Intent comments on every non-trivial change
  - README updated as part of this phase, not after
  - Both normaliseCampaignJson files updated together if either changes

After completing all changes, post a comment on issue ISSUE_NUMBER:

  ---
  *🤖 This comment was generated by Claude Code*
  ---

  ## Phase 3 complete: Build

  **Changes made:**
  | File | What changed |
  |---|---|
  [one row per file created or modified]

  **Deviations from plan:**
  [anything that differed from the Phase 2 plan and why — or "None"]

  **Comments added:**
  [confirm that intent comments were added to all non-trivial changes]

Do not proceed to Phase 4 until the comment is posted.

---

PHASE 4 — TEST

Step 4a — Run automated tests

Run any test files created or identified in the Phase 2 plan.
Report the exact terminal output. All automated tests must pass
before proceeding. If any fail, fix the code and re-run.

Step 4b — Static verification

Without running a browser, verify each item in the Phase 2 test
strategy that is marked "automated" or is a structural check
(file exists, script load order, function signature, etc.).
Report pass or fail for each.

Step 4c — Manual test instructions

If the Phase 2 test strategy includes any tests marked "manual",
print clear step-by-step instructions for the DM to follow using
Live Server in VS Code. Each manual test must include:
  - Numbered steps to reproduce
  - Exactly what to look for (console, network tab, UI behaviour)
  - Explicit pass criteria

If there are no manual tests, state: "No manual tests required."

Post a comment on issue ISSUE_NUMBER:

  ---
  *🤖 This comment was generated by Claude Code*
  ---

  ## Phase 4 complete: Test

  **Automated test results:**
  [paste exact terminal output]

  **Static verification:**
  | Check | Result |
  |---|---|
  [one row per check]

  **Manual tests:**
  [either the step-by-step instructions, or "None required"]

Do not proceed to Phase 5 until all automated tests pass and the
comment is posted.

---

PHASE 5 — CLOSE

Step 5a — Final checklist

Read the acceptance criteria from the issue body. Verify each one
is met. Report pass or fail for every criterion. If any criterion
is not met, fix it before continuing.

Step 5b — Post completion comment on issue ISSUE_NUMBER

  ---
  *🤖 This comment was generated by Claude Code*
  ---

  ## Work complete — issue ISSUE_NUMBER

  ### Summary
  [2-3 sentences describing what was done and why]

  ### Files changed
  | File | Change |
  |---|---|
  [one row per file]

  ### Test results
  - Automated: [X/Y passed]
  - Manual: [instructions provided for DM review / None required]

  ### Acceptance criteria
  [copy each criterion from the issue and mark [x] or [ ]]

  ### Notes for dependent issues
  [Read the issue body for any issues listed as depending on this
  one. If any exist, describe observations from this work that are
  relevant to them. If none are listed, write "None identified."]

Step 5c — Close or hand off

Determine whether any manual tests were documented in Phase 4.

IF manual tests exist:

  DO NOT close the issue.

  1. Add the label "in-review" to issue ISSUE_NUMBER in GitHub
  2. Post a final comment:

      ---
      *🤖 This comment was generated by Claude Code*
      ---

      ## Awaiting manual review

      All automated tests passed. Manual testing is required before
      this issue can be closed.

      **Manual test instructions are in the Phase 4 comment above.**

      When you have completed the manual tests:
      - All tests pass → close this issue manually in GitHub
      - Any test fails → add a comment describing what failed;
        the issue remains open for follow-up

  3. Print this message to the terminal and stop:

      ⏸  AWAITING MANUAL REVIEW
      ─────────────────────────────────────────────
      Issue ISSUE_NUMBER has been labelled "in-review".
      Manual test instructions are posted on the issue.

      Complete the tests in Live Server, then close the issue
      manually in GitHub when satisfied.

      This session is complete. Do not make further changes.

  Do not close the issue. Session ends here.

IF no manual tests exist:

  Close issue ISSUE_NUMBER in GitHub.

  Print this message to the terminal and stop:

      ✅  ISSUE ISSUE_NUMBER COMPLETE
      ─────────────────────────────────────────────
      All tests passed (automated only).
      Issue ISSUE_NUMBER has been closed.

      Check the project board for the next issue to work on.

  Session ends here.

---

STANDING RULES (apply throughout all phases)

- Read before writing. Never modify a file you haven't read in full.
- Comment intent, not mechanics. Every non-trivial change needs a
  comment explaining why, not just what.
- README stays current. Any change that affects described behaviour
  must update README.md in the same phase — not after.
- Post phase comments before moving on. Each phase comment is a
  checkpoint. If you cannot post it, stop and report why.
- One issue at a time. Do not begin any other issue during this
  session, regardless of what dependent issues are listed.
- If a test fails, fix it before proceeding. Do not close an issue
  with failing tests.
- Never self-close an issue that has outstanding manual tests. If
  manual tests exist, label "in-review" and stop. Closing is the
  DM's responsibility after manual verification is complete.
- Do not assume which issue comes next. The project board determines
  priority — do not reference or begin any other issue number.
```

## Reusing this prompt for future issues

Change only the four lines in the declaration block:

```
ISSUE_NUMBER:  [the issue number]
REPO_OWNER:    dgrimesii
REPO_NAME:     Chronicle-MagersCampaign
ISSUE_URL:     https://github.com/dgrimesii/Chronicle-MagersCampaign/issues/[the issue number]
```

Nothing else changes. The five phases, standing rules, comment
formats, gate, and close procedure are all generic and apply to
every issue without modification.
